# 用 Go + eBPF 构建蜂群式网络防御系统：7层管线 + 64核锁自由评分器

> 一个能单机每秒评分 3200 万次的开源防御系统是如何设计的

## 引言

传统的网络防御系统（IDS/IPS）面临三个核心问题：

1. **检测层单一**——大多数系统只有 1-2 层检测机制，容易绕过
2. **性能瓶颈**——软件包处理在高速网络下扛不住
3. **情报孤岛**——每个节点各自为战，没有协作

我花了一年多时间，用 Go + Rust + eBPF 写了一个叫 **Fortress** 的系统，试图同时解决这三个问题。

这个项目已经开源：
**GitHub: [github.com/hyson810/fortress](https://github.com/hyson810/fortress)**

这篇文章不是产品介绍，而是**架构决策记录**——为什么这么设计，踩了什么坑，学到了什么。

---

## 一、为什么是 7 层管线？

大部分 IDS 只有一层检测：抓包 → 规则匹配 → 告警。

问题在于：单一检测层无法覆盖所有攻击面。

举个例子：
- 一个 DNS 隧道工具，包级别看完全是合法的 DNS 查询
- 但从**行为层面**看，每分钟几千次 TXT 查询到同一个域，明显异常
- 从**流量层面**看，每个响应包都超大，不符合 DNS 的正常分布

所以我把检测拆成 7 层：

```
L1: 包检测（Packet Inspector）      — eBPF/XDP 层面，特征匹配
L2: 流分析（Flow Analyzer）         — 5元组 + 统计特征
L3: 行为分析（Behavior Analyzer）    — 基线画像偏离检测
L4: DNS隧道检测（DNS Tunnel Detector）— 熵值 + 流量特征
L5: HTTP检测+暴力破解               — 请求分析 + 频率检测
L6: 混合异常检测（ML + 启发式）       — 模型 + 规则
L7: 指纹引擎（JA3/SHA256/HASSH）     — TLS/SSH 指纹识别
```

每层把检测结果传给下一层，最后汇聚到评分器。

**这样设计的好处：**
- 单层绕过不影响整体判断
- 每层可以做自己最擅长的事（L1 快而糙，L6 慢而准）
- 组合特征比单一特征难逃逸得多

**代价：** 延迟叠加。实测从包到达 → 输出评分，中位数延迟约 120μs（裸 XDP 约 5μs）。不是零开销，但对大多数场景是可以接受的。

## 二、64 核锁自由评分器

这是系统里最"轴"的一个模块。

### 为什么需要锁自由？

威胁评分是**热点操作**——每个包、每秒钟几百万次访问。如果用 mutex 保护评分状态，核心数一旦上去，锁争用就会把性能吃掉。

典型的 Go 并发模型是 goroutine + channel，但在 3200 万次/秒的吞吐下，channel 的开销也是不可接受的。

### 怎么做到的？

核心思路是 **Per-CPU sharding**：

```
评分状态 → 分片到 64 个 shard
每个 shard 只在自己的 CPU 上操作
读取时汇总所有 shard（原子操作）
```

每个 shard 内部用 **atomic 操作**代替锁：

```go
type Shard struct {
    counters [1024]atomic.Uint64
    threshold atomic.Int64
}
```

`Store` 和 `Load` 都是无锁的。只有在读取全局状态（比如"当前总威胁分数"）时才需要一次 O(n) 的 shard 遍历。

**单机实测结果：** 约 3200 万次评分/秒，0 额外内存分配。

### 一个有意思的优化

评分器里有一个"热度衰减"机制——如果某个 IP 在一段时间内没有新事件，它的分数会指数衰减。这个衰减如果用定时器扫全表，在 64 核下会触发大量的缓存失效。

解法：**不做定时扫表**，而是在**读取时惰性计算**衰减。每个分数存一个时间戳，读的时候 `score * math.Exp(-elapsed * decayRate)`。精确度略有损失，但避免了全局扫表。

## 三、SWIM 流行协议：去中心化情报网格

这是我对现有防御系统最不满意的地方——大部分系统是中心化的。

中心化的问题：
1. **单点故障**——SIEM 挂了，所有节点变盲
2. **带宽瓶颈**——几千个节点的日志全往中心送
3. **延时**——从检测到情报同步到其他节点，分钟级别

### 为什么选 SWIM

SWIM（Scalable Weakly-consistent Infection-style Membership Protocol）是去中心化节点发现和状态传播的协议。

优点：
- **无中心节点**——每个节点都是对等的
- **最终一致**——状态通过 gossip 传播，几秒内全网收敛
- **可扩展**——理论上支持数千节点

### 我们加的东西

原版 SWIM 只做成员管理，我们在上面加了三层：

1. **威胁情报广播**——检测到威胁时，立刻通过 gossip 扩散给所有对等节点
2. **Raft 共识决策**——当需要执行"封禁"这种不可逆操作时，先用 Raft 达成共识
3. **免疫传播**——如果一个节点确认了对某个威胁的免疫（比如更新了规则），免疫状态会像疫苗一样在网格里传播

### 实战效果

5 个节点组成的蜂群，一个节点检测到扫描 → 1.2 秒内其他 4 个节点全部更新封禁规则。而传统 SIEM 模型，从检测到推送到所有节点通常在 30 秒以上。

## 四、自适应蜜罐

蜜罐是一个老概念，但传统蜜罐有几个硬伤：

1. **容易被识别**——标准的 SSH 横幅，nmap 一扫就知道是蜜罐
2. **静态指纹**——跑久了攻击者会总结规律
3. **容易被 DDoS**——没做限流的蜜罐就是别人免费的攻击目标

### 动态指纹化

我们的蜜罐（SSH/HTTP/MySQL）每次启动时**随机生成服务指纹**：

- SSH：随机选择 banner 版本 + 密钥交换算法组合
- HTTP：随机选择 Server header + 响应顺序
- MySQL：随机生成版本字符串 + 认证插件列表

这样每次扫描结果都不同，攻击者无法建立"这是蜜罐"的特征库。

### 减速带

蜜罐检测到可疑交互后，**渐进式增加响应延迟**：

```
第 1 次连接: 正常延迟 (5ms)
第 5 次连接: 500ms 延迟
第 10 次连接: 5s 延迟
第 15 次连接: 30s 延迟
```

目的不是阻止，而是**消耗攻击者的时间**。自动化工具在蜜罐上卡住 30 秒，是很好的拖延战术。

### 数据收集

所有蜜罐交互完整记录，包括：
- 完整的攻击命令序列
- 尝试的凭据组合
- 下载的 payload（沙箱隔离）

这些数据可以用来更新检测规则，也可以作为威胁情报的输入。

## 五、C2/Dagger 框架（为什么防御系统需要攻击模块？）

这是最容易被误解的部分。Fortress 包含了一个完整的 C2 框架（代号 Dagger），包括：

- **Rust 植入体**：支持进程注入、持久化、横向移动
- **多协议监听器**：DNS / HTTPS / WebSocket / ICMP / SMB
- **Payload 编译器**：动态生成免杀 payload
- **混淆器**：代码混淆 + 流量加密

**为什么防御系统需要这些？**

很简单：**不理解攻击，就无法有效防御**。

Dagger 框架在 Fortress 里的定位是：

1. **自测**——你在部署防御系统之前，先用自己的 C2 框架打一遍自己，看看防御能不能拦住
2. **红队自动化**——授权渗透测试中，用 Dagger 做攻击模拟
3. **威胁狩猎**——用植入体特征反推攻击者的 TTP

当然，这部分代码是双刃剑。**仅用于授权测试和教育目的。**

## 七、踩过的坑

### 1. Go 的 GC 和 XDP 不搭

在 XDP 程序里分配 Go 对象会导致 GC 压力暴涨。方案：**预分配 ring buffer + 零拷贝传递**。

### 2. Rust 植入体编译时间

C2 框架的 Rust 植入体，完整编译一次 4-7 分钟。开发体验很糟糕。方案：用 cargo-chef 做 Docker 分层缓存，CI 里从 7 分钟降到 45 秒。

### 3. SWIM 的脑裂问题

早期版本中，网络分区恢复后，SWIM 网格会出现分裂——两边的节点互不承认。方案：引入 Raft 做关键操作的共识层，SWIM 只做成员发现和状态传播，不做决策。

### 4. 蜜罐被当成攻击目标

部署蜜罐的第一周，就被 Shodan 扫描器发现了，然后变成了僵尸网络的攻击靶子。方案：蜜罐部署在独立网络命名空间里 + 严格的速率限制。
